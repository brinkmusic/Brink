# Feature: text-only posts for users and artists (T104)

## Rationale
Right now Brink only lets you post *media*: a regular user must attach a song, and an
artist must attach a photo, before they can share anything. People asked to be able to
"just write something" — a plain text post — like every other social app. This removes the
media requirement on both post types while keeping the media *optional*, so a post can be:
a song + text, a song only, text only (users); or a photo + text, a photo only, text only
(artists). A post with neither media nor text is still rejected — you can't share nothing.

## Summary
Make the song optional on regular posts (`Post.trackId`) and the photo optional on artist
posts (`ArtistPost.imageUrl`), so both can be **text-only**. Both composers gain an
always-visible text box + Share; attaching a song / photo becomes an optional step. The feed,
profile, and artist pages render a distinct **note-style card** for text-only posts (no album
art / play button / image). A server-side guard rejects a post that has neither media nor text.

## Current State (verified 2026-07-23)
- **Data model** (`backend/app/models.py`): `Post.trackId` is a **NOT NULL** FK to
  `silver.Track.spotifyId`; `Post.caption` is already nullable. `ArtistPost.imageUrl` and
  `ArtistPost.caption` are both **NOT NULL**.
- **Create endpoints**: `POST /api/posts` (`routers/posts.py`) always `upsert_track` +
  links `track_id`. `POST /api/artist/posts` (`routers/artist.py`) always sets `image_url`.
- **Feed / list queries**: `_build_song_items` (`routers/feed.py`) and `list_posts`
  (`routers/posts.py`) **INNER JOIN** `Track` — a post with no track would silently vanish.
  `_build_artist_items` signs `image_url` via `create_signed_read_url_or_blank` (T103).
- **Schemas** (`schemas.py`): `CreatePostBody.track` required; `PostOut.track` /
  `FeedPostOut.track` required. `CreateArtistPostBody.image_url` + `.caption` required;
  `ArtistPostOut` / `ArtistFeedPostOut` carry required `image_url` (+ `caption`).
- **Composers**: `feed.html` + `composer.js` reveal caption + Share only **after** a track
  is picked. `artist.html` + `artist-upload.js` keep the caption box **hidden** until a valid
  image is chosen (this was the deliberate T57 behavior) and require a caption to share.
- **Rendering**: `feed.html` branches on `p.kind` ("song" vs "artist"); the artist card
  already distinguishes signed-URL vs blank via the T103 placeholder (`artist-post-img-missing`).

## Scope
### In Scope
- **Migration**: `Post.trackId` → nullable; `ArtistPost.imageUrl` → nullable;
  `ArtistPost.caption` → nullable. Autogenerated, verified with `alembic check`. Applied to
  brink-dev (and prod at release).
- **Guard (both endpoints)**: reject (400) a post with **no media and no non-empty text**.
  Regular post: needs a `track` OR a non-blank `caption`. Artist post: needs an `image_url`
  OR a non-blank `caption`.
- **Backend endpoints**: `POST /api/posts` accepts a body with no `track` (text-only,
  `track_id = NULL`, `source` still supplied — MANUAL for typed text). `POST /api/artist/posts`
  accepts no `image_url`.
- **Feed / list**: LEFT (outer) join `Track` in `_build_song_items` and `list_posts` so
  text-only posts appear; build `track` only when present; skip the T102 play-count for
  trackless posts. Artist feed: `image_url = None` for text-only (distinct from `""` =
  signing failed, T103).
- **Schemas**: `track` / `image_url` become `Optional`; response DTOs allow `None`.
- **Composers**:
  - *User* (`feed.html` + `composer.js`): caption text box + Share always visible; song search
    + "🎧 Share what you're hearing" become an optional attach step; a selected song shows as a
    removable chip; Share posts text + optional track; empty (no text, no song) is blocked
    with a status message.
  - *Artist* (`artist.html` + `artist-upload.js`): caption box always visible (revert the T57
    hide); Share enabled when there's a valid image **or** non-blank caption; upload path runs
    only when an image is picked, otherwise it posts caption-only.
- **Note-style rendering** (chosen look): text-only user + artist posts render a distinct
  note card (larger/quote-style text, tinted background) — `feed.html`, `artist.html`,
  `brink.css`. A song/photo post is unchanged.
- **Tests**: create text-only post (user + artist) round-trips; empty post rejected (400);
  feed includes a text-only post and omits track/play-count; list_posts returns a trackless
  post; page/markup + composer.js/artist-upload.js source assertions for the new flows.

### Out of Scope
- Rich text, mentions, hashtags, links — plain text only (same `caption` field, existing
  length limits).
- Multiple photos / photo galleries; video.
- Any change to reactions/comments (they already work per-post regardless of media).
- Backfilling or migrating existing posts (all existing posts have media; nothing to move).

## Outcome (as built)
- **Migration** `fe4e66a13f08`: relaxes NOT NULL on `Post.trackId`, `ArtistPost.imageUrl`,
  `ArtistPost.caption` (additive/backward-compatible — no data moves). Applied to brink-dev
  (dev == prod today); `alembic check` clean.
- **Schemas** (`schemas.py`): `CreatePostBody.track`, `CreateArtistPostBody.image_url`/`.caption`
  now `Optional`; `PostOut`/`FeedPostOut.track`, `ArtistPostOut`/`ArtistFeedPostOut.image_url` +
  `.caption` allow `None`. `FeedPostOut.kind == "song"` now means "a regular user post" (documented,
  not renamed).
- **Endpoints**: `POST /api/posts` (`routers/posts.py`) upserts a track only when one is sent and
  writes `trackId = NULL` otherwise; `POST /api/artist/posts` (`routers/artist.py`) accepts no
  image. Both trim the caption and return **400 `"a post needs a song/photo or some text"`** when a
  post has neither media nor text. `list_posts` LEFT-joins Track.
- **Feed** (`routers/feed.py`): `_build_song_items` LEFT-joins Track, builds `track` only when
  present, and skips the T102 play-count for trackless posts; `_build_artist_items` sets
  `image_url` **tri-state** — a signed URL, `""` (had an image, signing failed → T103 placeholder),
  or `None` (text-only → note). `pages.py` mirrors this via a shared `_artist_image_url` helper and
  guards the flatten against a `None` track.
- **Composer** (`feed.html` + `composer.js`, `?v=104`): the caption textbox + **Share** are always
  visible; the song search + "🎧 Add what you're hearing" (T101) are an optional attach step; a
  picked track shows as a removable chip (`composerRemoveTrack`); Share publishes text + optional
  song and blocks a fully-empty share client-side.
- **Artist upload** (`artist.html` + `artist-upload.js`, `?v=104`): caption always visible (reverts
  T57's reveal-on-image); `artistCaptionInput` enables Share on text alone; the sign+upload runs
  only when a file is picked, otherwise it posts caption-only.
- **Note styling** (`brink.css`): text-only user (`.post-note`/`.post-note-body`) and artist
  (`.artist-post-note`/`-note-body`) posts render a distinct note card — tint + accent edge, larger
  quote-like text, no album-art/play button/image.
- **Tests**: `test_posts.py` (text-only create + empty 400 + trackless list), `test_artist.py`
  (photo-only / text-only create + empty 400, replacing the old caption-required test), `test_feed.py`
  (feed includes text-only user + artist posts), `test_pages.py` (composer/artist-upload source +
  note-card render + caption-always-visible).
- **Deliberately out of scope:** rich text/mentions/hashtags/links, multiple photos/video, and any
  reactions/comments change (already per-post regardless of media).

## Notes / Risks
- **Shared prod DB** (T99): the migration must be applied to brink-dev (dev == prod today).
  It only *relaxes* NOT NULL, so it's safe and backward-compatible — existing rows and the old
  media-required flow keep working.
- The `FeedPostOut.kind == "song"` discriminator now means "a regular user post" (may or may
  not have a track), not literally "has a song". Documented in the schema comment rather than
  renamed (a rename would ripple through templates/JS for no behavior gain).
- Artist feed `image_url` is now tri-state: a signed URL (show image), `""` (had an image,
  signing failed → T103 placeholder), or `None` (text-only → note card). Template distinguishes
  `None` from `""`.
- Frontend touches Sebastian's page layer — flag him for review on the template/JS/CSS parts.
